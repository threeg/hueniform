This brief scopes the next version of Hueniform. It is the v0.2.0 analogue of the original
project brief and, once approved, drives a set of **requirement deltas** against the living
specification rather than a fresh spec. The binding documents in `docs/` are not forked per
version: they evolve in place, and `git` (the `v0.1.0` tag) is the record of what shipped.
New requirements take new `FR-`/`NFR-` numbers; superseded ones are amended in place and noted,
never silently reinterpreted (CLAUDE.md; `CONVENTIONS.md` §5.5).

---

## 1. Aim of v0.2.0

v0.2.0 is a **product-feature** release — a grouped batch of the *essential* capabilities the app
needs before the advanced roadmap items (weather-aware suggestions, wear tracking, hosting) are
considered. Those roadmap items (v0.1.0 brief §10) remain **out of scope** for v0.2.0.

The meta-goal is unchanged: the process is a deliverable. v0.2.0 also adopts, as *how* we build
rather than as a goal in itself, the **test-first** policy agreed for deterministic and
contract-pinned layers (see §7).

The local-first contract is preserved: **every feature below runs offline** with no runtime network
calls. NFR-1 and NFR-8 are unaffected (this is the main reason weather-aware suggestions are
deferred — they need an online data source).

## 2. Feature set

Seven feature areas (F1–F7), grouped into epics (§4). For each: what it is, the requirement
impact (new vs changed), the layers it touches, and its risk.

### F1 — Build an outfit around a chosen garment

Let an outfit request **pin a specific garment** to its slot ("suggest around this jacket / these
shoes"); the suggester returns harmonious outfits that all include it, and fails fast with a clear
message if none exist.

- **Requirements:** new (proposed **FR-44**). Extends **FR-36** (request options).
- **Layers:** `matcher` (enumeration must honour a fixed garment), `services` (suggestion),
  `api` (suggestions request schema), `frontend` (outfit-request screen).
- **Risk:** low–medium. Builds on the existing enumeration and fallback ladder (FR-39–FR-43).

### F2 — Build an outfit around a colour scheme

Let a request **anchor to colour** — a target colour family ("an outfit around teal") and/or a named
harmony scheme (analogous, complementary, …); suggestions are constrained to satisfy it.

- **Requirements:** new (proposed **FR-45**). Extends **FR-36**; reuses the scheme definitions in
  **FR-12–FR-14**.
- **Layers:** `matcher`, `services`, `api`, `frontend`.
- **Risk:** medium. Needs a precise definition of "around a colour" (family target vs named scheme
  vs both) — an open question for the requirements pass (§8).

### F3 — Edit a garment's category

Allow the **type (category)** of a saved garment to be edited directly, without re-running detection.
Colour/palette changes remain **regenerate-only** (FR-33). Metadata (brand, season, notes) and an
availability flag were considered but are **deferred to a future version** — not in v0.2.0.

- **Requirements:** new (proposed **FR-46**). **Amends FR-32**: the "not field-editable" rule is
  narrowed so the *type* field is editable; palette stays regenerate-only. FR-25/FR-26 unchanged.
- **Layers:** `services`, `api`, `frontend` (garment detail). No `storage` schema change.
- **Risk:** low. A single editable field; suggestion eligibility simply follows the new type.

### F4 — Category & slot-model overhaul (the central workstream)

The garment-type taxonomy and the rigid outfit structure are **redesigned**, not merely extended.
The intent from the interview:

1. **Overhaul the categories** — a revised, fuller set of garment types.
2. **Default, removable slots** — today every outfit *must* contain top + bottom + socks + shoes
   (FR-17). v0.2.0 keeps a sensible default set **selected by default** (the current four) but lets
   the user **deselect** any of them per request — e.g. drop the top and socks for a beach outfit.
   The exact default set, and whether any minimum viable outfit applies, come out of the spike.
3. **Mutual-exclusion rules** — some categories conflict and cannot co-occur. Examples: *pants* vs
   *shorts* vs *skirt* (one lower-body item at a time); a one-piece *dress*/*jumpsuit* occupies the
   upper- **and** lower-body region and therefore excludes a separate top and bottom.

This is **not a list change** — it is a redesign of the slot/anchor model that the matcher's harmony
evaluation is built on (FR-16–FR-22: anchors, dominant layer, echo slots). It is the largest and
highest-risk piece of v0.2.0 and is treated as its own epic, **led by a design spike** before any
implementation ticket (§4, §6).

- **Requirements:** substantial. New: proposed **FR-49** (revised taxonomy), **FR-50** (slot groups
  with mutual exclusion), **FR-51** (configurable required slots), and a one-piece model. **Rewrites
  FR-16, FR-17, FR-18, FR-19, FR-20, FR-21, FR-22.** The concrete rules are produced by the spike and
  written into `docs/02-requirements.md` §5 before implementation.
- **Layers:** `matcher` (taxonomy, slots, roles, harmony anchors — the 100%-covered core),
  `services`, `api`, `frontend`.
- **Risk:** **high.** It refactors the deterministic matcher. Mitigations: golden-file snapshot tests
  pinning current matcher output *before* the refactor (per `docs/meta/method-improvements.md` #2),
  and test-first on the new rules.

### F5 — Suggestion quality (diagnostic-led)

This is broader than randomisation and is driven by concrete v0.1.0 observations; all of it stays in
v0.2.0 (no patch release). A diagnostic spike over the current `suggestion_service`, `matcher.ranking`
and taxonomy will separate genuine defects from tuning, then we refine the rules. Three threads:

1. **Diversity.** Returned outfits are near-duplicates (one or two swaps on the same base), and
   genuinely different valid outfits stop surfacing as the wardrobe grows — consistent with the
   candidate-enumeration cap (NFR-5) being reached before diverse candidates are enumerated, plus
   ranking dominance. Strengthen the variety factor (**FR-41**) and revisit enumeration order.
2. **Neutrals never surface.** An all-neutral outfit (e.g. all black) is harmonious under the
   neutral-based scheme (FR-13, order 1), yet never appears, because it scores lowest on scheme
   strength and FR-43 returns neutral-based only as a *fallback*. Decide and encode whether
   all-neutral outfits should be first-class results (likely refine **FR-41/FR-43**).
3. **Pale near-neutrals misclassified.** Cream/ecru (e.g. the white jeans) is *measured* correctly
   but *classified* as Yellow: it falls outside both White (L>92 **and** S<20) and Beige/Tan (S≤45,
   L≤88), so it lands in the Yellow arc. No family covers "pale warm near-neutrals." Fix by tuning
   the **FR-2** taxonomy table (a §1.4 constants change, possibly a new off-white/cream family).

- **Requirements:** refine **FR-41, FR-42, FR-43**; amend the **FR-2** taxonomy; new **NFR-10** —
  variety must be **seedable** so the pure matcher stays unit-testable (preserves NFR-9).
- **Layers:** `matcher` (taxonomy, ranking), `services`. May touch detection fixtures for thread (3).
- **Risk:** medium. Determinism vs variety resolved by a seeded RNG (fixed seed in tests, free at
  runtime). Coordinate with E08, which also touches the matcher.

### F6 — Inventory grouping & ordering

Replace today's flat, filter-only inventory with garments **grouped by category**, and within each
group orderable by **hue — as a colour spectrum (the default)** — or by **date added, newest
left-to-right**. (Search, multi-sort and multi-photo are **not** wanted.)

- **Requirements:** new (proposed **FR-47**). Extends **FR-35** (the existing type/colour filters
  remain). Hue ordering uses each garment's primary-colour hue.
- **Layers:** `frontend` (grouped layout + ordering); optional `api`/`services` ordering support.
  No `storage` schema change. NFR-6 (responsive at 500 garments) still applies.
- **Risk:** low.

### F7 — Configurable number of suggestions

Raised alongside F6, but it belongs to suggestion output: today a request returns **up to 3** outfits
(FR-39). Let the user choose how many to generate — **default 3, maximum 25** — as a setting shown
*before* clicking Generate.

- **Requirements:** **amends FR-39** ("up to 3" → "up to N, user-selected, 1–25, default 3"). New
  (proposed **FR-48**) for the request parameter and its bounds.
- **Layers:** `services`, `api` (request schema + validation), `frontend` (the setting).
- **Risk:** low mechanically, but it **raises the stakes on F5**: asking for 25 makes today's
  diversity weakness obvious, so F5 and F7 should land together. NFR-5 (≤2 s at 500 garments) must
  still hold at the higher count, or the cap and performance budget be revisited.

## 4. Proposed epics

Continuing the `HUE-E0N` scheme (next is **E06**):

| Epic | Title | Features | Notes |
|---|---|---|---|
| **HUE-E06** | Constrained suggestions | F1, F2 | Pin-a-garment and anchor-to-colour. Depends on the F4 model being settled, since "a slot" and "a scheme" are redefined by it. |
| **HUE-E07** | Edit a garment's category | F3 | Small and additive; independent of E08, can proceed early. |
| **HUE-E08** | Category & slot-model overhaul | F4 | **Foundational and highest-risk.** Led by a design spike; rewrites the matcher slot/anchor model. Sequence first. |
| **HUE-E09** | Suggestion quality & count | F5, F7 | Diagnostic-led (diversity, all-neutral outfits, near-neutral taxonomy). F7's 1–25 count forces the F5 diversity fix, so they ship together. Coordinate with E08's matcher changes. |
| **HUE-E10** | Inventory grouping & ordering | F6 | Mostly frontend; independent of the matcher. |

**Sequencing.** E08 is on the critical path: it redefines slots and garment types, which E06 and E09
build on. Recommended order is **E08 (spike → requirements → implementation) → E06/E09**, with **E07
and E10 run in parallel** since they barely touch the matcher.

Implementation tickets continue the monotonic scheme from the current maximum: next is **HUE-059**.

## 6. Build approach (carried from method discussion)

- **Test-first** for the deterministic and contract-pinned layers (`matcher`, `services` logic,
  `api` contract shapes): write the failing test from the requirement/contract, confirm it fails for
  the right reason, then implement. **Characterisation/after** tests remain appropriate for the
  probabilistic `detection` layer.
- **Snapshot the matcher before E08.** Add golden-file tests pinning current ranking, classification
  and explanation output, so the slot-model refactor surfaces any behavioural change in the diff.
- **Spike-first for E08.** A time-boxed `spike` ticket produces the new slot/exclusion model and the
  FR-16–22 rewrite *before* implementation tickets are written.
- **Diagnostic spike for E09.** A `spike` over the current `suggestion_service`, `matcher.ranking`
  and taxonomy separates genuine defects (lost diversity, suppressed all-neutral outfits, the
  cream→Yellow gap) from tuning, before the F5/F7 tickets are written.

## 7. Open questions for the requirements pass

These do not block approval of this brief; they are settled in milestones 10–11.

1. **The exact category list** and the mutual-exclusion groups (lower-body group; one-piece group;
   how layering — jersey/jacket — interacts with one-piece items). [F4]
2. **Minimum viable outfit** — given the default slots are removable, is a too-sparse (or empty)
   selection rejected, and what is the floor? [F4]
3. **"Build around a colour"** — target colour *family*, a *named scheme*, or both? [F2]
4. **All-neutral outfits** — should they be first-class suggestions, and how are they ranked relative
   to chromatic schemes? [F5]
5. **Near-neutral taxonomy** — tune the existing White / Beige-Tan thresholds, or add a dedicated
   off-white/cream family? [F5]
6. **Count vs performance** — does NFR-5 (≤ 2 s at 500 garments) hold at 25 requested outfits, or is
   the enumeration cap / performance budget revisited? [F7]

## 8. Risks

| Risk | Notes / mitigation |
|---|---|
| E08 refactors the 100%-covered matcher | Snapshot baseline before refactor; test-first on new rules; keep the spike's model fully specified before coding. |
| Scope creep across five epics | Sequence E08 first; run E07/E10 in parallel; treat E06/E09 as dependent on E08. Each ticket still scoped to one Code session. |
| Suggestion-quality work spans ranking, neutral policy and taxonomy (F5) | Lead with the diagnostic spike to separate defects from tuning; seedable RNG (NFR-10) keeps variety testable. |

| Local-first contract | Preserved — no v0.2.0 feature needs the network; weather deferred for exactly this reason (NFR-1/NFR-8 intact). |
